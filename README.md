# Темзит MQTT Bridge — аддон Home Assistant

Локальный мост **TCP ↔ MQTT** для тепловых насосов (гидромодулей) ТЭМЗИТ.
Аддон напрямую опрашивает гидромодуль по его собственному протоколу (TCP, порт **333**)
и публикует параметры в Home Assistant через **MQTT Discovery** — без облака
`service.temzit.ru`.

Репозиторий: <https://github.com/daemon707/temzit-addon>
Текущая версия аддона: **0.8.1**

> **Старые версии (≤0.5.0)** ходили в облако Темзит. Начиная с 0.7.x аддон работает
> **только локально** по реверс-инженеренному протоколу гидромодуля. Облачные
> логин/пароль больше не нужны.

## Что умеет

**Чтение (разобрано и подтверждено до байта по реальным дампам + веб-интерфейсу):**

- Температуры: улица, в доме, подача, обратка, фреон газ/жидкость, ГВС (знаковые, ÷10).
- Состояние: режим (СТОП/НАГРЕВ/ГВС/ТЭН), активность и частота компрессора, проток,
  ступень ТЭНа, нагреватель БКН, потребляемая мощность.
- Авария: 32-битное поле с расшифровкой кодов `E..` (включая ошибки второго ККБ).
- Версия прошивки, часы контроллера (BCD), день недели.
- Активное расписание: номер, режим, уставки дома/воды/ГВС, режим ТЭНа/ГВС, ограничение ККБ.
- Конфигурация (CONFIG_MAIN, ~16 параметров ядра): уставки, режим ТЭНа/ГВС, внешний котёл,
  ограничение мощности ККБ, погодокомпенсация, импульсы электросчётчика, тип расходомера и др.

**Запись — с защитой.** Команда `0x35` (CFG) реализована (режим, уставки дома/воды/ГВС,
ограничение ККБ) и обёрнута тремя предохранителями: **обязательный накопительный бэкап**
рабочего дампа перед записью (нет бэкапа → запись отменяется), **дедупликация** (не писать без
реального изменения — беречь flash) и **валидация результата** (нереалистичные значения, ранее
роняшие контроллер, отклоняются). Подробнее — раздел «Запись» ниже.

## Сущности в Home Assistant

Создаются автоматически через MQTT Discovery с фиксированным `obj_id`, поэтому
`entity_id` стабильны: `sensor.temzit_<ключ>`.

- `climate.temzit_climate` — отопление (уставка дома, режим off/heat/cool).
- `climate.temzit_dhw_climate` — ГВС (уставка ГВС).
- ~29 сенсоров: температуры, проток, мощность, частота компрессора, состояние, авария,
  версия ПО, уставки конфига и активного расписания.

Готовый дашборд — [`docs/temzit_lovelace_card.yaml`](docs/temzit_lovelace_card.yaml)
(вставить в HA как **Manual card**).

## Установка

**Вариант 1 — как репозиторий аддонов (рекомендуется):**

1. Home Assistant → **Settings → Add-ons → Add-on Store → ⋮ → Repositories**.
2. Добавить URL: `https://github.com/daemon707/temzit-addon`.
3. В магазине появится **Temzit MQTT Bridge** — установить.

**Вариант 2 — локально:**

1. Скопировать папку `temzit_mqtt_bridge/` в `/addons/` на вашем HA.
2. Supervisor → Add-on Store → **Локальные аддоны** → **Temzit MQTT Bridge** → установить.

После установки заполнить параметры (вкладка **Configuration**) и запустить аддон.

## Параметры конфигурации

| параметр | по умолчанию | назначение |
|---|---|---|
| `temzit_host` | — | IP гидромодуля (например `192.168.2.20` в LAN или `192.168.4.1` в сети ГМ) |
| `temzit_port` | `333` | TCP-порт гидромодуля |
| `temzit_timeout` | `15` | таймаут TCP-операций, с |
| `temzit_sync_interval` | `60` | период опроса состояния (SYNC), с — **не чаще 10 с** |
| `temzit_cfg_interval` | `900` | период опроса конфигурации (REQCFG), с (`0` — не опрашивать) |
| `temzit_cfg_delay_after_sync` | `12` | пауза между SYNC и REQCFG, с (порт держим коротко) |
| `temzit_retry_delay` | `15` | пауза перед повтором при ошибке/занятом порте, с |
| `data_dir` | `/share/temzit` | каталог для накопительных бэкапов конфига перед записью |
| `mqtt_host` | — | адрес MQTT-брокера |
| `mqtt_port` | `1883` | порт брокера |
| `mqtt_user` / `mqtt_pass` | — | учётные данные брокера (если включена авторизация) |
| `mqtt_prefix` | `temzit` | префикс топиков состояния/команд |
| `mqtt_discovery_prefix` | `homeassistant` | префикс MQTT Discovery |
| `mqtt_client_id` | `temzit-bridge` | client id для брокера |

> При `host_network: true` имя `core-mosquitto` может не резолвиться — укажите **IP** брокера.

## Эксплуатационные ограничения протокола

- Опрашивать **не чаще раза в 10 с**. TCP держать коротко: connect → 1 запрос → ответ → close.
- Порт бывает занят (WiFi-модуль выгружает статистику в облако) — аддон делает ретраи.
- **Ресурс циклов записи во flash ограничен** → команда CFG должна слаться только при реальном
  изменении настройки.

## Диагностика

[`docs/temzit_report.py`](docs/temzit_report.py) — автономная **read-only** утилита (только
стандартная библиотека Python 3, не зависит от моста). Снимает два SYNC с паузой (видно, какие
байты меняются), один REQCFG, печатает полный hex-дамп, проверку CRC16 и расшифровку:

```sh
python3 docs/temzit_report.py --host 192.168.2.20
python3 docs/temzit_report.py --host 192.168.2.20 --port 333 --out my_report.txt
```

Удобно гонять прямо с NAS/HAOS или из SSH-терминала аддона. Запись (`0x35`) скрипт не выполняет.

## Протокол (кратко)

Гидромодуль — TCP-сервер на порту 333. Кадр: `type (1 байт) + body`. Три документированные команды:

| команда | запрос | ответ | размер |
|---|---|---|---|
| SYNC `0x30` | `0x30 0x00` | ACTUAL_STATE `0x01` | 64 байта |
| REQCFG `0x34` | `0x34 0x00` | CONFIG_MAIN `0x02` | 64 байта |
| CFG `0x35` (запись) | `0x35` + 30 байт + 1 байт КС | ACTUAL_STATE | 32 байта |

- В ответах (`0x01`/`0x02`) КС — **16-битная** сумma байт `data[:62]`, лежит в `data[62:64]` (LE).
  Полезный массив начинается с `data[2]` (есть байт-пэддинг `data[1]=0x00`).
- В записи (`0x35`) КС — **1 байт**, сумма всех байт включая командный.

Полная спецификация — [`docs/HM_Protocol.pdf`](docs/HM_Protocol.pdf) (последняя версия с
temzit.ru). Детальные раскладки байт ACTUAL_STATE и CONFIG_MAIN — в `CLAUDE.md`.

### Что недоступно локально

Локальный протокол отдаёт только **ядро** конфигурации (~16 параметров). Ряд параметров
(P10, P15/P16 гистерезисы, P60–P64 солнечный коллектор, типы ККБ2, полная таблица из 4 строк
расписания и др.) физически отсутствуют в ответе `0x34` — они существуют только в облачном
приложении `service.temzit.ru`, где перевод «P-код → байт» делает сервер. Через локальный
протокол доступно только **активное** расписание.

> ⚠️ Недокументированные команды перебирать **нельзя** — PDF предупреждает о риске повредить
> контроллер.

## Запись

Команда CFG реализована с предохранителями (см. ниже). Низкоуровневые `build_setcfg`/`set_cfg`
(КС 1 байт, read-modify-write всех 30 байт, WiFi-байты 10–16 не трогаются) не менялись.

**Что происходит при каждой команде записи:**

1. Применяем изменение к копии текущего конфига и сверяем **результат**.
2. **Дедуп:** если ничего не поменялось — запись пропускается (`diag/set_guard: nochange`).
3. **Валидация результата** `looks_like_valid_cfg(new_cfg)` — нереалистичные значения
   отклоняются (`diag/set_guard: rejected`), на контроллер ничего не уходит.
4. **Обязательный бэкап** рабочего дампа: уникальный `cfg_<timestamp>.json` + строка в
   накопительный `temzit_cfg_history.jsonl`. Не удалось записать бэкап → запись **отменяется**
   (`diag/set_guard: backup_failed`).
5. Только теперь отправляется `0x35`, путь к бэкапу попадает в `diag/last_set`.

**Куда пишутся бэкапы.** По умолчанию `/share/temzit` (опция `data_dir`, env `TEMZIT_DATA_DIR`) —
этот каталог доступен пользователю через Samba/File editor/SSH, в отличие от `/`. Если он
недоступен — выбирается первый записываемый из `/share/temzit → /config/temzit → /data/temzit →
./temzit_backups`. Бэкапы **накапливаются** и никогда не перезаписываются, так что всегда есть
история рабочих дампов для отката.

> Восстановление из бэкапа пока ручное: в файле есть `cfg_raw`/`cfg_hex` (30 байт), их можно
> вернуть побайтно командой `cmd/set_byte`. Автоматический «restore» — кандидат на следующую итерацию.

**Команды MQTT** (топики относительно `mqtt_prefix`): `climate/set_mode`,
`climate/set_temperature`, `climate/set_water_temp`, `climate/set_dhw_temp`,
`climate/set_compressor_limit`, `cmd/set_byte` (диагностический: `{"offset": N, "value": V}`).

**Ещё не закрыто** (следующие итерации): маскированная запись нибблов для offset 3/6/19/23 и
режима ГВС в младший ниббл offset 6; защита `cmd/set_byte` от записи в WiFi-байты 10–16;
автоматический restore из бэкапа.

## Технические детали

- База образа: Alpine 3.19, `py3-paho-mqtt` (paho API v1). **Не поднимать базу до 3.20+** без
  перевода MQTT-клиента на callbacks paho v2.
- HA фиксирует `entity_id` при первом создании. Чтобы переименовать сущности: очистить retained
  discovery в MQTT, удалить сущности, перезапустить аддон.

Историю изменений см. в [`CHANGELOG.md`](CHANGELOG.md).
